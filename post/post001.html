2. git status

terminal을 열어서 local directory로 이동한 다음 먼저 git status를 이용해서 현재 상태를 확인해준다.
나는 지금 작업할 main 브랜치랑 github page용 gh-pages 브랜치 두개밖에 없고, 진짜 블로그하는 것만 우선 목적이라서 내가 main에 있는지 확인해주는 용도로 사용함.
git status 명령어를 수행하면 결과로 'modified: 작업물' 이런 결과가 나온다. 아 코드 표시 어떻게 하지? 저번에 찾아보고 공부했었는데 생각이 안 남... 암튼, modified 색깔이 빨간색이면 add를 해줘야 한다.(add는 바로 밑에) modified 색깔이 초록색이면 이제 commit해도 됨!


Tip. 1번 작업은 main 브랜치에서 해야 한다! 근데 다른 브랜치에서 작업했으면 메인으로 이동해서 재작업하거나 (아직 테스트 안 해봄: 아님 다른 브랜치를 메인에다 복사해주면 되지 않을까?)
이거 쓰면서 gh-pages 브랜치에서 작업해서 강제 테스트해봄ㅋㅋㅋ큐ㅠㅠㅠ 결론은 그러면 된다.
Tip. git status는 자주자주 확인해보는 습관을 들이자. 마치 ctrl + s 를 습관화하는 것처럼...!!!


3. git add

git add는 지금 작업중인 working directory의 파일을 staging area로 옮겨주는 역할을 한다. 로컬에서 작업한 작업물을 원격 저장소로 옮기려면,

  1. 로컬에서 작업을 하고
  2. 작업물을 staging area로 옮기고
  3. 작업물을 커밋하고(로컬 저장소에 저장함)
  4. 그걸 푸시하고(로컬저장소에서 원격저장소로 업로드)
  5. 필요시 브랜치를 만들고 브랜치들을 서로 복사하거나 병합하거나 한다.
그래서, git add 는 커밋하기 전에 해줘야 하고, 'git add .' 을 입력하면 지금 있는 디렉토리에서 안 옮겨진 모든 파일을 다 staging area로 옮겨준다.
git add . 를 수행하고 다시 git status를 입력하면 modified 부분이 초록색으로 변한 걸 확인할 수 있다. 그럼 이제 커밋하면 된다.


4. git commit

커밋은 내가 이해한 바로는 쉽게 말하면 세이브파일같은건데, 문서작업할 때 수없이 생기는 엑셀파일들처럼

  1. 작업물 ver1
  2. 작업물 ver1.1
  3. 작업물 ver1.2
  4. 작업물 ver2
뭐 이런거나 또는
  1. 작업물_최종
  2. 작업물_최종_진짜최종
  3. 작업물_최종_진짜최종_최종2222
뭐 이런 걸 생각하면 대충 비교가 되지 않을까 싶다. 여기서 첫번째가 좋은 네이밍의 예시고 두번째가 나쁜 네이밍의 예시가 되겠지ㅋㅋㅋ
그래서, commit을 하면, 로컬에서 마지막으로 저장한 폴더의 상태가 저장된 엑셀파일 하나가 만들어진다고 보면 된다.
아 이건 근데 깃 개념이해 뭐 이런걸로 포스팅 따로 해야겠다.. 뭐가 많네 비교표도 넣고싶고.. 일단 지금은 커밋만 하자.
git commit -m "작업한 내용" 명령어를 입력하면 "작업한 내용"을 이름으로 가지는 커밋이 생성된다. -m의 m은 message의 약자이다.


5. git push

커밋을 했다. 그럼 이제 푸시해서 원격 저장소로 업로드하자. 로컬은 내가 작업하고 있는 내 컴퓨터 환경을 말하고 원격은 인터넷상에 있는 작업물을 올려 저장할 공간이다.
처음에 원격저장소가 생성되면 origin이라는 닉네임이 디폴트로 붙는다. 내가 어떤 원격저장소를 가지고 있는지 확인해보려면 git remote 명령어를 입력하면 된다. git push origin을 입력해서 리모트 레파지토리(원격 저장소)에 푸시해주자.


6. git branch

git branch를 사용하면 브랜치 목록이 나온다.


7. git checkout

git checkout은 내가 작업할 브랜치로 위치를 변경해준다. git status를 입력해서 내가 main에 있다는 걸 확인하고, git checkout gh-pages 를 입력하여 gh-pages 브랜치로 이동해준다.


8. git merge

두 개의 브랜치를 병합해준다. 나는 병합한다고 해서 병합된 브랜치가 사라지는 줄 알았는데 그건 아니더라. 오히려 복사에 가까운듯. A 브랜치와 B 브랜치를 똑같은 상태로 만들어준다고 생각하면 되는 듯하다.
내가 A 브랜치에 있는 걸 B 브랜치로 머지하고 싶으면 커맨드는 B 위치에서 git merge A 를 입력한다. 나같은경우는 main에서 작업하고 gh-pages로 복사해주는 거고, 그래서 git checkout gh-pages로 내 위치를 이동한 후 git merge main 을 입력해주면 된다. A랑 B에서 같은 코드에 대해 서로 다르게 변경하고 머지하려고 하면 컨플릭트가 발생한다는데 나는 작업용 main이랑 깃헙페이지용 gh-pages 두개밖에 없어서 지금은 그럴 일은 없을듯.
그럼 이제 gh-pages에 main이 그대로 옮겨졌으니 git push origin 을 입력해서 gh-pages를 리모트 레포에 업로드해주면 된다! 그럼 끝!
Tip. 끝나고나면 다시 git checkout main 을 입력해서 main branch(작업할 곳)로 돌아가주는 걸 잊지 말자.
Tip. github desktop 앱에서 제공하는 branch - Updage from master 기능과 동일한 결과가 나온다.


9. git log (q 눌러서 exit)

git log 는 말 그대로 내가 git을 이용해 수행한 로그를 보여준다. 앞서 말했듯 나는 github desktop application 을 이용하는 방식에 익숙해 있었는데, 깃헙데스크탑에서는 Update from master 기능을 이용한다. main이라는 이름의 브랜치가 마스터브랜치로 설정되어 있어서 그런건데, 거기에 익숙해진 나는 git merge master 를 입력했고, 머지할 게 없다는 결과가 나왔다.
응?? 싶어진 나는 로그를 보려고 git log 를 입력했고 뭐지? 뭐지? 하면서 git status를 입력해보려는데 커맨드가 안 쳐졌다. 띠로리...

이건 캡쳐하려고 다시 입력한거라서 당시 상황이랑 좀 다른데, Initial commit 밑에 : 이렇게만 되어있길래 구글에 initial commit, git log initial commit 등을 검색해보면서 initial commit이 뭔지 설명하는 글만 잔뜩 읽고서 겨우 해답을 찾았다.
q키를 누르면 git log를 빠져나올 수 있다!!
아니 저번에 git log 써봤을 때는 아무 문제 없었는데 왜 이번엔 그랬는지 (그리고 테스트해본다고 한번더랑 캡쳐한다고 한번더 해봤을 때도) 오늘은 지쳤으니 나중에 찾아보겠지만 아무튼 q를 누르고 해방되었다. 이얏호! (*저거 가지고 해방이라느니 유난같지만 난 프렙 과제한다고 매우 지쳐있었다ㅠㅠ)
